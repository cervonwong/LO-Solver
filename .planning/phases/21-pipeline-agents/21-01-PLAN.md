3. **Task**: Parse the raw problem text into structured markdown. Describe the three sections to extract:

   - **Dataset**: Complete sentence pairs only. Renumber sequentially (#1, #2...). Auto-detect which column is the foreign language vs. English based on content. Handle multi-language problems with 2+ languages and variable column counts -- do not assume a fixed two-column format. Items with missing translations (blanks, "?", "___") go to Questions, not Dataset.
   - **Questions**: Both explicit questions ("Translate...") and implicit questions (items with blanks). Renumber sequentially (Q1, Q2...). Classify direction (e.g., "Foreign -> English", "English -> Foreign"). Copy question text EXACTLY as written.

4. **Output Format**: Embed the `problem.md` template from workspace-format.md directly. Show the exact markdown structure:

   ```
   ## Dataset
   | # | {Language1} | {Language2} |
   |---|-------------|-------------|
   | 1 | ... | ... |
   ## Questions
   | # | Direction | Sentence |
   |---|-----------|----------|

   ```
   Note: Column headers use actual language names when known, not "Foreign" / "English".

   - Do NOT include items with missing translations in the Dataset -- they belong in Questions
   - Do NOT modify, correct, or "fix" any text -- copy everything verbatim from the input

Replace the placeholder body in `hypothesizer.md` with a complete self-contained system prompt. PRESERVE the existing YAML frontmatter exactly as-is.

**Prompt structure (in this order):**

1. **Domain Context** (2-3 sentences): Explain what a Rosetta Stone problem is and what this agent discovers (linguistic rules and vocabulary mappings from an assigned analytical perspective).

2. **Input**: The agent receives (a) the path to `problem.md` (the extracted problem -- describe what it contains so the agent knows what to look for), (b) an assigned perspective with name + focus description, (c) the round number and perspective number, (d) the output file path, and optionally (e) baseline rules/vocabulary to build upon for Round 2+ (or start from scratch if not provided).

3. **Task**: Analyze the problem from the assigned perspective. Discover vocabulary mappings (individual morpheme-to-meaning pairs) and grammatical rules (patterns and mechanisms). Keep vocabulary and rules strictly separate.

4. **Analysis Process** (7 steps, adapted from Mastra hypothesizer):
   1. **Segmentation and Alignment** -- Break phrases into smallest meaningful units, align foreign segments with English counterparts across all examples, create a mental translation table
   2. **Identify Recurring Patterns** -- Find segments appearing multiple times, note which English concepts map to which foreign segments, track word order differences
   3. **Morphological Analysis** -- Identify affixes (prefixes, suffixes, infixes, circumfixes), root words, agglutination patterns, agreement patterns (number, gender, case, person)
   4. **Syntactic Analysis** -- Determine word order (SVO, SOV, VSO, etc.), modifier placement, question formation, negation
   5. **Phonological Analysis** (if applicable) -- Sound changes, spelling variations, vowel harmony
   6. **Validation and Falsification** -- Test each rule against EVERY example. Actively seek counterexamples. If a rule doesn't hold, question whether it is correct at all. Only add exceptions if they are simple and follow Occam's Razor
   7. **Generate Competing Hypotheses** -- For each significant pattern, generate at least one alternative interpretation. Compare based on (a) examples explained, (b) simplicity, (c) predictive power. Choose best and briefly note why alternatives were rejected

5. **Output Format**: Embed the `perspective-N.md` template from workspace-format.md directly:
   ```
   # Perspective {N}: {Perspective Name}

   ## Vocabulary

   | Foreign Form | Meaning | Type | Notes |
   |-------------|---------|------|-------|
   | {morpheme} | {english gloss} | {type} | Appears in #{sentence numbers} |

   ## Rules

   ### 1. {Rule title}

   {Description of the pattern/mechanism}

   **Evidence:** #{sentence numbers}

   **Confidence:** {HIGH|MEDIUM|LOW}
   ```
   Vocabulary types: noun, verb-root, adjective, pronoun, tense-marker, case-marker, particle, affix, etc.

6. **Confidence Guidelines** (from Mastra pipeline, proven through evaluation):
   - **HIGH**: Explicitly checked every relevant example and found no contradictions. Pattern is unambiguous and simple. State: "Checked against #X, #Y, #Z -- no contradictions found."
   - **MEDIUM**: Rule is complex or has edge cases suggesting the formulation may need refinement, OR not yet verified against all relevant examples.
   - **LOW**: Hypothesized based on analogy or intuition. Evidence is weak or ambiguous. May need significant revision.

7. **Do NOT** section (anti-patterns):
   - Do NOT test or verify your rules against the dataset by attempting translations -- testing is the verifier's job (separate agent), to avoid confirmation bias
   - Do NOT include vocabulary items in your Rules section -- vocabulary maps individual morphemes to meanings; rules describe patterns and mechanisms (e.g., "verbs take -ti suffix for past tense"), not lexical entries (e.g., "boro = fish")
   - Do NOT reference other perspectives' outputs -- you work in complete isolation
   - Do NOT answer the questions -- only discover rules needed to answer them
   - Do NOT assume rules from similar real-world languages unless the dataset evidence supports them
   - Do NOT write rules without citing specific dataset sentence numbers as evidence
   - Do NOT write vocabulary entries without citing which sentences they appear in (via the Notes field)
   - Do NOT conflate tense markers, case markers, or agreement markers with root vocabulary -- they are separate morphemes
   - Do NOT mark a rule as HIGH confidence unless you have explicitly verified it against every relevant example

8. **Error Handling**: If the problem.md cannot be read or is malformed:
   - Write a partial output file explaining what could be analyzed and what failed
   - Append an entry to `errors.md` in the workspace run folder with: Time, Agent (hypothesizer, perspective-N), Error type, Message, Recovered status

**Key adaptation notes from Mastra hypothesizer:**
- The Mastra hypothesizer commits vocabulary/rules via CRUD tools; this agent writes them as markdown sections in a single output file
- The Mastra hypothesizer has testRule/testSentence tools for self-validation; this agent does NOT self-test (locked decision from CONTEXT.md) -- the verifier agent handles testing separately
- Preserve the 7-step analysis process (proven through evaluation), the confidence guidelines, and the vocabulary/rules separation principle
- The Mastra hypothesizer receives perspective as a JSON object with `linguisticAngle`; this agent receives it as plain text (name + focus description) in the orchestrator's message
- Add the evidence citation requirement from CONTEXT.md: every rule MUST cite specific sentence numbers, every vocabulary entry MUST note which sentences it appears in
