  <files>src/contexts/workflow-control-context.tsx</files>
  <action>
Create `src/contexts/workflow-control-context.tsx` following the MascotContext pattern but with a "register" approach since the state originates from a child component (page.tsx) rather than being defined in the provider itself.

The context value interface:
```typescript
interface WorkflowControlContextValue {
  isRunning: boolean;
  hasStarted: boolean;
  stop: () => void;
  handleReset: () => void;
}
```

Implementation approach:
1. Create `WorkflowControlProvider` that holds state via `useState` for `isRunning` and `hasStarted`, and `useRef` for `stop` and `handleReset` callbacks (refs avoid stale closures).
2. Create `useWorkflowControl()` hook â€” consumes the context, throws if used outside provider (same pattern as `useMascotState`).
3. Create `useRegisterWorkflowControl(opts: { isRunning: boolean; hasStarted: boolean; stop: () => void; handleReset: () => void })` hook â€” called by page.tsx to push its state into the context. Uses `useEffect` to sync `isRunning`/`hasStarted` boolean state. Uses `useEffect` to keep the `stop`/`handleReset` ref callbacks current.

The provider exposes the context value by reading from its internal state and wrapping ref callbacks in stable functions. This way nav bar components call `stop()` and it delegates to whatever page.tsx registered.

Default values (before page.tsx registers): `isRunning: false`, `hasStarted: false`, `stop: noop`, `handleReset: noop`.

Mark file with `'use client'` directive.
