**Domain Context:** The verifier is a mechanical tester, not a creative analyst. It tests one thing per call -- either checking a rule's correctness or performing a blind translation of a sentence.

**Input section** (4 items):
1. Test type -- either "rule" or "sentence" (the /solve skill tells the verifier which mode)
2. Path to the current solution file (solution.md or improved-{N}.md) -- the verifier reads rules and vocabulary from here
3. Path to problem.md -- for dataset reference
4. For rule tests: the specific rule title to test. For sentence tests: the specific sentence to translate (foreign text + direction)
5. Output file path -- where to write the single test result

**Task section** with two modes:

### Rule Test Mode
- Read the solution file to get all rules and vocabulary
- Read problem.md to get the dataset
- Find the specific rule by title in the solution
- Check the rule against ALL dataset examples: Is it correct? Consistent? Sufficient?
- Look for counterexamples, edge cases, and conflicts with other rules
- Report status: PASS, FAIL, or NEEDS_UPDATE
- Include reasoning with specific dataset citations

### Sentence Test Mode (Blind Translation)
- Read the solution file to get all rules and vocabulary
- Read the sentence to translate (provided by the /solve skill)
- Translate using ONLY the rules and vocabulary from the solution file
- Apply rules systematically: identify morphemes, apply morphological rules, apply syntactic rules, construct translation
- Return ONLY the blind translation -- do NOT compare it to any expected answer
- The /solve skill will do the comparison externally

**Output Format:**
- For rule tests: Write a short markdown file with: Rule title, Status (PASS/FAIL/NEEDS_UPDATE), Reasoning (2-3 sentences citing dataset examples), Recommendation (if not PASS)
- For sentence tests: Write a short markdown file with: Sentence (the input), Translation (the blind translation), Working (brief morpheme breakdown showing how rules were applied)

**Do NOT section** (at least these items):
- Do NOT compare your blind translation to any expected answer (the /solve skill handles comparison)
- Do NOT aggregate multiple test results (you test ONE thing per call)
- Do NOT modify or improve rules (that is the improver's job)
- Do NOT answer questions -- only test what you are given
- Do NOT read any file other than the solution file and problem.md
- Do NOT mark a rule as PASS unless you have checked it against all relevant dataset examples

**Error Handling:** Follow the established pattern -- write partial output + append to errors.md.

Reference the Mastra verifier-orchestrator instructions (03a) for the testing logic, but adapt for single-test-per-call design. The Mastra verifier tests everything in one call; the Claude Code verifier tests ONE thing per call.

### improver.md

Set `model: opus` in frontmatter (per user decision -- rule improvement requires creative linguistic reasoning).

Follow the exact section structure from hypothesizer.md.

**Domain Context:** The improver is a Reviser PhD linguist. It reads verification failures, performs root cause analysis, and produces a revised complete solution (vocabulary + rules).

**Input section** (4 items):
1. Path to current solution -- solution.md or improved-{N}.md (the /solve skill tells the improver which file to read)
2. Path to verification results -- verification.md or verification-{N}.md
3. Path to problem.md -- for dataset reference
4. Output file path -- improved-{N}.md (where N is the current iteration number)

**Task section** -- adapt from the Mastra rules-improver instructions (03b-rules-improver-instructions.ts):
1. Read all three input files (current solution, verification results, problem.md)
2. Analyze the verification failures: which rules failed? Which sentences failed? What are the root causes?
3. Perform root cause analysis following the 6 core reasoning principles from the Mastra improver:
   - Logical order: update vocabulary first if the issue involves word meanings
   - Abductive reasoning: consider wrong morpheme segmentation, phonological rules, null morphemes
   - Multiple hypotheses: generate at least one alternative fix
   - Adaptability: discard rules if evidence shows they were wrong
   - Grounding: quote exact examples from the verification
   - Persistence: mentally test revised rules before committing
4. For each failing rule: analyze the failure, generate alternative hypotheses, select the best fix, cite evidence
5. For each failing sentence: trace back to which rule(s) caused the failure
6. Check for missing rules (patterns no existing rule explains)
7. Write the complete revised solution (not just diffs -- a full replacement file)

**Output Format:** Same structure as solution.md:
```
# Improved Solution (Iteration {N})

## Changes
{Bulleted summary of what changed and why}

## Vocabulary
{Full vocabulary table -- same format as solution.md}

## Rules
{Full rules sections -- same format as solution.md, with updated/added/removed rules}
```

Include a `## Changes` section at the top summarizing what was changed and why (root cause analysis summary). Then include the COMPLETE vocabulary table and rules, not just the changed ones.

**Confidence Guidelines:** Same as hypothesizer (HIGH/MEDIUM/LOW with evidence requirements).

**Do NOT section:**
- Do NOT only fix the symptoms -- perform root cause analysis
- Do NOT preserve rules just because they existed before -- discard if evidence contradicts
- Do NOT produce partial output -- write the COMPLETE vocabulary and rules
- Do NOT use vocabulary items in rules or rules in vocabulary
- Do NOT ignore any issue from the verification -- address every failure
- Do NOT answer the questions

**Error Handling:** Established pattern.

### answerer.md

Set `model: opus` in frontmatter (per user decision -- systematic derivation requires complex reasoning).

Follow the exact section structure from hypothesizer.md.

**Domain Context:** The answerer is a systematic translator applying validated rules. It reads the final solution and problem, then translates each question with full derivation steps.

**Input section** (3 items):
1. Path to final solution -- solution.md or latest improved-{N}.md
2. Path to problem.md -- for questions and dataset reference
3. Output file path -- answers.md

**Task section** -- adapt from the Mastra question-answerer instructions (04-question-answerer-instructions.ts):
For each question in problem.md:
1. Identify the task type (translate-to-english, translate-to-foreign, fill-in-blank, error-correction)
2. Parse the input: break into component morphemes/words
3. Apply rules systematically: vocabulary lookup, morphological rules, syntactic rules. Track which rules are applied.
4. Construct the answer from rule application
5. Document working steps: morpheme segmentation, rule-by-rule application with citations, interlinear gloss lines, synthesis

Per user decision: include confidence level + reasoning for each answer (HIGH/MEDIUM/LOW). Per user decision: ALWAYS produce a best-attempt translation, flag with LOW confidence and explain what is uncertain. NEVER leave a question unanswered.

**Output Format:** Follow the workspace-format.md answers.md template exactly:
```
# Answers

## Q{N}: {Direction}

**Input:** {sentence}
**Translation:** {answer}
**Confidence:** {HIGH|MEDIUM|LOW}

**Working:**
- {morpheme breakdown}
- {rule application with citations}
- {interlinear gloss}
- {synthesis}

**Rules applied:** {comma-separated rule titles}
```

**Do NOT section:**
- Do NOT use external linguistic knowledge -- only the provided rules
- Do NOT skip any question -- always produce a best-attempt translation
- Do NOT invent rules not in the solution file
- Do NOT omit working steps -- every answer needs full derivation
- Do NOT read any file other than the solution file and problem.md

**Error Handling:** Established pattern. If a question truly cannot be answered, still produce a best-attempt with LOW confidence explaining what is uncertain.

Reference the Mastra answerer instructions (04) for the methodology, but adapt for file-based I/O and markdown output (not JSON).
