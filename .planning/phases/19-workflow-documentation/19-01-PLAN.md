Create the `claude-code/` directory and write `claude-code/PIPELINE.md`.

**Document structure** (pipeline-sequential, per CONTEXT.md decisions):

1. **Overview** -- Pipeline summary paragraph. Quick-reference table of all agents (ID, model, role, tools). Brief data flow diagram showing Step 1 → Step 2 (multi-round loop) → Step 3 → Step 4.

2. **Shared Concepts** -- Cross-cutting patterns used by multiple agents:
   - **Two-Agent Chain pattern**: Reasoning agent (Gemini 3 Flash) generates natural language analysis → extraction agent (GPT-5-mini) parses into structured JSON. Used in hypothesis extraction, feedback extraction, and improvement extraction. Include 1-2 sentence design rationale (why separate models, why split reasoning from extraction).
   - **Vocabulary system**: All 5 CRUD tools (`addVocabulary`, `updateVocabulary`, `removeVocabulary`, `getVocabulary`, `clearVocabulary`), the `VocabularyEntry` schema, how vocabulary is shared across agents via shared state, and the prompt fragment pattern (agents receive vocabulary tool instructions injected into their system prompt). Design rationale: why vocabulary is separate from rules.
   - **Rules system**: All 5 CRUD tools (`addRules`, `updateRules`, `removeRules`, `getRules`, `clearRules`), the `Rule` schema.
   - **Testing tools**: `testRule`/`testSentence` (committed variants for verifiers) vs `testRuleWithRuleset`/`testSentenceWithRuleset` (draft variants for hypothesizers/synthesizer). Explain the blind translation pattern: sentence tester translates without seeing the expected answer, comparison happens post-hoc. Design rationale for committed vs draft variants.
   - **Draft store isolation**: Each perspective operates on isolated rules/vocabulary. After verification, a synthesis step merges the best results. Framework-agnostic description (no RequestContext).
   - **Core data types**: Inline TypeScript interfaces for `Rule`, `VocabularyEntry`, `StructuredProblem`, `VerifierFeedback`, `QuestionAnswer`, `Perspective`. Use simplified clean interfaces from 19-RESEARCH.md Code Examples section -- NOT raw Zod definitions.

3. **Step 1: Extract Structure** -- Agent: `structured-problem-extractor` (GPT-5-mini). Light treatment per CONTEXT.md: "parses raw problem text into structured JSON." Include the `StructuredProblem` schema. Describe input (raw problem text) and output (structured problem JSON). This is a single-agent step, no two-agent chain.

4. **Step 2: Multi-Perspective Hypothesis Generation** -- The most complex section. Document the multi-round loop structure:

   **Round structure** (for round = 1 to maxRounds):
   - **(a) Dispatch**: Round 1 uses `perspective-dispatcher` agent. Round 2+ uses `improver-dispatcher` agent (receives current rules, vocabulary, test results, previous perspective IDs as additional input). Both output a list of `Perspective` objects. Include the actual linguistic perspective fields (id, name, linguisticAngle, reasoning, priority). Summarize the dispatcher prompt: analyze the problem, identify linguistic dimensions worth exploring, generate diverse non-overlapping perspectives. Summarize the improver-dispatcher prompt: gap analysis of current rules vs test failures, generate perspectives targeting weaknesses.
   - **(b) Hypothesize (parallel)**: For each perspective, create an isolated draft store. Round 2+: initialize draft from current main store. Run `initial-hypothesizer` agent with the perspective's linguisticAngle injected. Detailed prompt summary for hypothesizer: systematic linguistic analysis, discover patterns, formulate rules, build vocabulary, test rules using draft tools, iterate until confident. Document tool access: vocabulary CRUD (5) + rules CRUD (5) + `testRuleWithRuleset` + `testSentenceWithRuleset`.
   - **(c) Verify (parallel)**: For each perspective, run `verifier-orchestrator` agent against that perspective's draft store. Verifier tests every rule via `testRule` and every dataset sentence via `testSentence`. Detailed prompt summary: systematic per-rule and per-sentence testing, what each granularity catches (per-rule: individual rule correctness; per-sentence: rule interaction and completeness). Then run `verifier-feedback-extractor` to get structured `VerifierFeedback`. Compute `testPassRate = 1 - (errantRules + errantSentences) / (rulesTestedCount + sentencesTestedCount)`.
   - **(d) Synthesize**: Programmatic vocabulary merge (iterate drafts worst-to-best by testPassRate, best perspective's entries win on conflict). Clear main rules store. Run `hypothesis-synthesizer` agent to merge rulesets. Synthesizer has access to vocabulary CRUD + rules CRUD + draft test tools. Detailed prompt summary: merge multiple perspective rulesets, resolve conflicts, use test tools to validate merged rules.
   - **(e) Convergence check**: Run `verifier-orchestrator` again on synthesized/merged main store rules. Run `verifier-feedback-extractor`. Check conclusion field. If `ALL_RULES_PASS` → converged, break loop. Otherwise clear draft stores, continue to next round.
   - **(f) Cleanup**: Clear all draft stores for next round.

   **Termination conditions** (all four):
   1. Convergence: verifier returns `ALL_RULES_PASS`
   2. Max rounds reached (default 3, testing mode: 2)
   3. Improver-dispatcher returns no perspectives (round 2+)
   4. Abort signal (external cancellation)

5. **Step 3: Verify & Improve (Legacy Path)** -- Note that in the current architecture, verification/improvement happens within Step 2's multi-round loop. Document the legacy `rules-improver` + `rules-improvement-extractor` two-agent chain briefly. These agents exist in the codebase but the primary improvement mechanism is the improver-dispatcher generating new perspectives. Describe the improver's role: receives `VerifierFeedback` (errant rules, errant sentences, issues list), revises rules and vocabulary. Light treatment.

6. **Step 4: Answer Questions** -- Agent: `question-answerer` (Gemini 3 Flash). Receives validated rules, vocabulary, structured problem, and questions. Detailed prompt summary: apply rules systematically to translate each question, show working steps, assign confidence. Output: array of `QuestionAnswer` objects. No tools -- pure reasoning.

7. **Design Rationale Summary** -- Table of key decisions and 1-2 sentence rationale each. Include: two-agent chains, blind translation, committed vs draft tools, per-perspective isolation, vocabulary separate from rules, multi-perspective + synthesis, convergence check after synthesis. Use the table from 19-RESEARCH.md.

**Writing guidelines** (from CONTEXT.md):
- Technical reference style: dense, structured, factual
- Sections with headers, bullet points, schemas
- Minimal prose -- every sentence carries information
- Framework-agnostic language ("shared state" not "RequestContext", "pipeline step" not "workflow step")
- Include brief design rationale per key decision (1-2 sentences)
- Primary audience: Claude Code Opus 4.6 agents

**Content extraction approach:**
- Read each `*-instructions.ts` file to extract the system prompt content
- For reasoning agents (hypothesizer, verifier orchestrator, synthesizer, dispatchers, answerer): write detailed prompt summaries capturing the key directives, not just a one-liner

- Read `workflow-schemas.ts` for schema definitions, simplify into clean TypeScript interfaces
- Read `vocabulary-tools.ts`, `rules-tools.ts` for tool signatures and behavior
- Read `vocabulary-tools-prompt.ts`, `rules-tools-prompt.ts` for the prompt fragment content (summarize, don't copy verbatim)

**Important accuracy notes from research:**
- CLAUDE.md describes 4 steps but code has 3 workflow steps. Present as 4 logical stages (matching user's mental model from CONTEXT.md) but explain that Steps 2 and 3 are interleaved in the multi-round loop.
- The `hypothesis-extractor` is DEPRECATED -- do not document its prompt in detail.
- Round 1 uses `perspective-dispatcher`, Round 2+ uses `improver-dispatcher` -- these are different agents with different inputs.
- There are TWO verification passes per round: per-perspective verification (scores individual hypotheses) and convergence verification (tests synthesized result). Both use the same `verifier-orchestrator` agent.
- The synthesizer does programmatic vocabulary merge (code, not agent) THEN agent-based rules merge. Document both.
