**Dispatcher agent** (`02-dispatcher-agent.ts`):
Create following the existing agent pattern (see interfaces). The dispatcher:
- ID: `perspective-dispatcher`
- Name: `[Step 2] Perspective Dispatcher Agent`
- Model: Gemini 3 Flash in production, TESTING_MODEL in testing (same dynamic model pattern as hypothesizer)
- No tools needed — the dispatcher only analyzes the problem and generates perspectives
- UnicodeNormalizer input processor (same config as all other agents)

**Dispatcher instructions** (`02-dispatcher-instructions.ts`):
Export `DISPATCHER_INSTRUCTIONS`. The prompt must instruct the agent to:

1. Analyze the structured problem data (context, dataset, questions)
2. Generate a set of linguistic perspectives to explore — each with a distinct angle
3. Include a REFERENCE LIST of common Linguistics Olympiad rule patterns to draw from:
   - Phonological: vowel harmony, consonant assimilation, tone patterns, sound changes
   - Morphological: agglutination, affixation (prefix/suffix/infix/circumfix), reduplication, ablaut
   - Syntactic: word order (SVO/SOV/VSO), modifier placement, question formation, relativization
   - Agreement: person/number/gender agreement on verbs, adjective agreement, case marking
   - Semantic: evidentiality markers, honorifics, classifier systems, tense/aspect/mood
   - Orthographic: special characters, diacritics, writing system conventions
4. Generate perspectives it thinks are feasible for THIS problem, including unlikely but possible ones
5. Each perspective must have: id (kebab-case), name, linguisticAngle (detailed exploration instructions), reasoning, priority
6. The number of perspectives is controlled by the caller (passed in prompt), not hardcoded
7. Output format: structured JSON matching dispatcherOutputSchema

The dispatcher does NOT use structuredOutput on the Agent — it will be called with `structuredOutput: { schema: dispatcherOutputSchema }` from the workflow step.

   - A "start over" perspective if the current approach is fundamentally flawed
5. Output: structured JSON matching improverDispatcherOutputSchema

**Synthesizer instructions** (`02-synthesizer-instructions.ts`):
Export `SYNTHESIZER_INSTRUCTIONS`. The prompt must instruct the agent to:

1. Receive all perspective results with their scores (passed in prompt as JSON)
2. The actual rules/vocabulary for each perspective are described in the prompt (serialized from draft stores by the workflow step)
3. Perform score-weighted conflict resolution:
   - When two rules explain the same phenomenon, favor the rule from the higher-scoring perspective
   - When rules from different perspectives are complementary (cover different phenomena), include both
   - When rules conflict, analyze which better fits the data and choose accordingly
4. Use rules CRUD tools to build the merged ruleset:
   - Call clearRules first to start fresh
   - Add winning rules via addRules
5. Use vocabulary CRUD tools to build merged vocabulary:
   - The workflow step handles programmatic vocabulary merging before the synthesizer runs (structured data merge)
   - Synthesizer may refine vocabulary entries if rule merging reveals issues
6. Validate the merged output using testRule/testSentence tools
7. The synthesizer MUST produce a coherent, non-contradictory ruleset
8. Include `{{RULES_TOOLS_INSTRUCTIONS}}` and `{{VOCABULARY_TOOLS_INSTRUCTIONS}}` placeholders
