**Rules CRUD tools** (`rules-tools.ts`):
Mirror the vocabulary-tools.ts pattern exactly. Create five tools using `createTool`:

1. `getRules` — Read all rules from the rules store in requestContext. Returns `{ rules: RuleEntry[], count: number }`.
2. `addRules` — Add new rules. Keyed by `title` (like vocabulary uses `foreignForm`). Skip duplicates. Returns `{ added, skipped, total }`.
3. `updateRules` — Update existing rules by title key. Skip if title not found. Returns `{ updated, skipped, total }`.
4. `removeRules` — Remove rules by title array. Returns `{ removed, notFound, total }`.
5. `clearRules` — Remove all rules. Returns `{ removed }`.

Use the existing `ruleSchema` from workflow-schemas.ts as the entry schema (title, description, confidence). The rules store is a `Map<string, Rule>` keyed by `title`.

Each tool must:
- Read the rules Map from requestContext using a `getRulesState` helper (like `getVocabularyState`)
- Emit trace events via `emitToolTraceEvent` with type `'data-rules-update'` and actions `add|update|remove|clear` (mirror the `data-vocabulary-update` pattern)
- Log operations via console.log with `[RULES:...]` prefix
- NOTE: Skip file-based logging (logRuleAdded etc) — the logging-utils does not need extension for rules at this time. The trace events are sufficient.

Export `rulesTools` object and `RulesTools` type, same pattern as `vocabularyTools`.

**Rules tools prompt** (`rules-tools-prompt.ts`):
Create `RULES_TOOLS_INSTRUCTIONS` constant mirroring `VOCABULARY_TOOLS_INSTRUCTIONS`. Include:
- Available tools (getRules, addRules, updateRules, removeRules, clearRules)
- Rule entry format (title, description, confidence)
- Usage guidelines: rules describe patterns/mechanisms, not vocabulary items
- Remind: "ALWAYS call getRules at the end of your task to verify rules state"

**Draft store infrastructure** (`request-context-types.ts`):
Add to `WorkflowRequestContext`:
- `'rules-state': Map<string, Rule>` — The main rules store (replaces `'current-rules'` for the new workflow; keep `'current-rules'` for backward compatibility with verifier tools)
- `'draft-stores': Map<string, DraftStore>` — Map of perspective ID to draft store

Define a `DraftStore` interface:
```typescript
export interface DraftStore {
  perspectiveId: string;
  vocabulary: Map<string, VocabularyEntry>;
  rules: Map<string, Rule>;
}
```

Keep all existing keys unchanged for backward compat.

**Draft store helpers** (`request-context-helpers.ts`):
Add these helper functions following the existing pattern (same RequestContextGetter type):

- `getRulesState(requestContext)` — Get the main rules Map, throws if not set
- `getRulesArray(requestContext)` — Get rules as array from Map
- `createDraftStore(requestContext, perspectiveId, pullFromMain: boolean)` — Creates a new DraftStore entry in the `draft-stores` Map. If `pullFromMain` is true, deep-copies the current main vocabulary and rules into the draft. Returns the DraftStore.
- `getDraftStore(requestContext, perspectiveId)` — Gets a specific draft store, throws if not found
- `getAllDraftStores(requestContext)` — Returns the full Map of draft stores
- `mergeDraftToMain(requestContext, perspectiveId)` — Copies the draft store's vocabulary and rules into the main stores (overwrites by key). Removes the draft store entry. Used by the synthesizer after selecting best rules.
- `clearAllDraftStores(requestContext)` — Clears the draft-stores Map

All helpers must handle the case where `'draft-stores'` is not yet initialized (create empty Map on first access).

For tools that need to operate on a specific draft store rather than main (used by hypothesizers), add:
- `getDraftVocabularyState(requestContext, perspectiveId)` — Returns the vocabulary Map from a specific draft store
- `getDraftRulesState(requestContext, perspectiveId)` — Returns the rules Map from a specific draft store

IMPORTANT: The existing `getVocabularyState`, `getCurrentRules`, etc. must continue to work unchanged — they read from main stores. Draft store access is a separate set of functions.
