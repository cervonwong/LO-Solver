**Workflow rewrite** (`src/mastra/workflow/workflow.ts`):
Replace `initialHypothesisStep` and `verifyImproveLoopStep` with a single new step: `multiPerspectiveHypothesisStep`.

Keep `extractionStep` (Step 1) UNCHANGED.
Keep `answerQuestionsStep` (Step 4) but modify it to read rules from the main `rules-state` Map in RequestContext instead of from the loop state. (The main rules store is populated by the synthesizer.)

**The new `multiPerspectiveHypothesisStep`:**

Input schema: `structuredProblemDataSchema` (from extraction step output, same as current initialHypothesisInputSchema)
Output schema: `questionAnsweringInputSchema` (structuredProblem + rules array — fed into answerQuestionsStep)
State schema: `workflowStateSchema`

Execute function logic:

```
1. Read maxRounds and perspectiveCount from state (set during extraction step from inputData)
2. Initialize RequestContext with main stores (empty vocabulary Map, empty rules Map, empty draft-stores Map)
3. For round = 1 to maxRounds:
   a. DISPATCH:
      - If round 1: call dispatcher agent with structured problem + perspectiveCount
      - If round 2+: call improver-dispatcher with structured problem + current main rules/vocab + previous test results
      - Parse response against dispatcherOutputSchema / improverDispatcherOutputSchema
      - Emit round-start event

   b. HYPOTHESIZE (parallel):
      - For each perspective:
        - Create draft store (pull from main if round > 1)
        - Create a RequestContext pointing to this perspective's draft stores:
          Set 'vocabulary-state' to draft vocabulary, 'rules-state' to draft rules
          (The hypothesizer's tools will read/write the draft via these keys)
        - Call hypothesizer agent with perspective-specific prompt:
          JSON.stringify({ structuredProblem, perspective, existingRules: round > 1 ? mainRulesArray : [], existingVocabulary: round > 1 ? mainVocabArray : [] })
        - Emit perspective-start / perspective-complete events
      - Use Promise.all to run all perspectives in parallel

   c. VERIFY (each perspective):
      - For each perspective's draft store:
        - Create a RequestContext with the draft rules as 'current-rules' and draft vocabulary as 'vocabulary-state'
        - Call verifier-orchestrator agent (uses testRule/testSentence tools from requestContext)
        - Call verifier-feedback-extractor to parse JSON
        - Compute test pass rate: 1 - (errantRules + errantSentences) / (rulesTestedCount + sentencesTestedCount)
        - Store perspectiveResultSchema data
      - NOTE: Verification can also run in parallel across perspectives

   d. SYNTHESIZE:
      - Sort perspectives by testPassRate descending
      - Programmatic vocabulary merge: iterate draft stores in score order, merge vocabulary Maps into main (last write wins by foreignForm key, higher-scored perspective's entries take priority)
      - Call synthesizer agent with: JSON of all perspective results + their rules (serialized from draft stores) + the programmatically merged vocabulary
        The synthesizer uses rules CRUD tools to build the merged ruleset in main store
      - After synthesizer completes, read main rules-state and vocabulary-state
      - Emit synthesis events

   e. CONVERGENCE CHECK:
      - Run verifier on the synthesized main rules one more time to get final score
      - If conclusion === 'ALL_RULES_PASS', break (early exit)
      - Store test results for potential round 2+ improver-dispatcher input
      - Emit round-complete event

   f. Clear all draft stores for next round

4. After all rounds: return { structuredProblem, rules: mainRulesArray }
```

IMPORTANT implementation notes:
- For parallel hypothesizer execution, each hypothesizer needs its OWN RequestContext instance pointing to its own draft store data. Do NOT share a single RequestContext across parallel calls — the 'vocabulary-state' and 'rules-state' keys must point to different Maps for each perspective.
- The simplest approach: for each perspective, create a fresh RequestContext and set 'vocabulary-state' to `draftStore.vocabulary`, 'rules-state' to `draftStore.rules`, 'structured-problem' to the shared problem data, 'log-file', 'model-mode', 'step-writer' to the shared values.
- Testing mode should use 2 perspectives and 2 max rounds (override perspectiveCount and maxRounds for testing mode). Production uses the values from input. Implement this as: `const effectivePerspectiveCount = state.modelMode === 'testing' ? Math.min(perspectiveCount, 2) : perspectiveCount;` and similar for maxRounds.
- The workflow chain becomes: `.then(extractionStep).map(...).then(multiPerspectiveHypothesisStep).then(answerQuestionsStep).commit()`
- Remove the `.dountil(verifyImproveLoopStep, ...)` chain entirely.
- Update `extractionStep` to also set `maxRounds` and `perspectiveCount` in state from inputData.
- In `answerQuestionsStep`, read rules from state.rules (Map serialized as record) instead of from inputData.rules, since the synthesizer writes to state. Alternatively, have multiPerspectiveHypothesisStep return rules in its output schema — this is cleaner. Keep using `questionAnsweringInputSchema` as the output of the hypothesis step.

Emit trace events throughout for frontend visibility. Use the step writer from the workflow step's `writer` parameter.
