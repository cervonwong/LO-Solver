**Detecting "aborted" state in page.tsx:**

When the user clicks Abort, `stop()` is called on the useChat hook. This sets `status` back to `'ready'`, so `isRunning` becomes false. However, `hasStarted` remains true, and the `workflowStatus` will NOT be `'success'` or `'failed'` — it will be whatever it was during execution (e.g., `'running'`), or potentially undefined if the stream was cut before a final workflow status event.

Define the aborted state:
```typescript
const isAborted = hasStarted && !isRunning && !isComplete && !isFailed;
```

This captures: the workflow was started, is no longer running, but did not complete successfully or fail — meaning it was user-aborted.

**Amber banner in page.tsx (inside the Progress card):**

After the existing `isFailed` error banner block (lines 642-658), add an `isAborted` banner:

Remove `hasStarted` from the solving condition — after abort, `isRunning` is false so mascot goes to idle. Per CONTEXT.md: "Mascot returns to idle state after abort."

**Progress bar freezes on abort:**

The progress bar already shows the current state based on `progressSteps` derived from workflow step data. When aborted, no more events arrive, so the progress bar naturally freezes. No code change needed for the freeze behavior.

Update the `statusMessage` derivation in page.tsx to handle the aborted case. Currently the message logic checks `workflowStatus`, then `activeStep`, then `status`. Add an aborted check:

After the `workflowStatus === 'failed' || workflowStatus === 'bailed'` condition, before the `activeStep` check, add:
```typescript
: isAborted
  ? 'Workflow aborted.'
```

This ensures the status message shows "Workflow aborted." in the progress panel.
