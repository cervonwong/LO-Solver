**3. Update `page.tsx`** to manage and expose `isAborting` state:

Add `isAborting` state:
```typescript
const [isAborting, setIsAborting] = useState(false);
```

Set `isAborting = true` when abort is initiated. The cleanest approach: detect when `isRunning` transitions from true to false while `isAborting` is true, then set `isAborting = false`. This means:
- The abort button in layout-shell calls `stop()` which changes the `useChat` status, making `isRunning` go false
- When `isRunning` goes false after aborting, clear `isAborting`

Add a `useEffect` to detect abort completion:
```typescript
useEffect(() => {
  if (!isRunning && isAborting) {
    setIsAborting(false);
  }
}, [isRunning, isAborting]);
```

But the aborting state needs to START when the user clicks abort. Since `stop()` is called from layout-shell via the context, the page needs to detect the transition. The approach:

a. Wrap the `stop` callback passed to `useRegisterWorkflowControl` so it also sets `isAborting`:
```typescript
const handleStop = useCallback(() => {
  setIsAborting(true);
  stop(); // the useChat stop
}, [stop]);
```

b. Pass `handleStop` instead of `stop` to `useRegisterWorkflowControl`:
```typescript
useRegisterWorkflowControl({ isRunning, hasStarted, isAborting, stop: handleStop, handleReset });
```

c. Clear `isAborting` when the workflow is no longer running (the abort has completed):
```typescript
useEffect(() => {
  if (isAborting && !isRunning) {
    setIsAborting(false);
  }
}, [isAborting, isRunning]);
```

d. Also clear `isAborting` in `handleReset`:
```typescript
const handleReset = useCallback(() => {
  // existing reset logic...
  setIsAborting(false);
  // ...
}, [/* deps */]);
```
