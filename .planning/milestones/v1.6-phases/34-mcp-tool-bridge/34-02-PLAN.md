  <action>
Add conditional MCP server and provider creation to each step file where tool-using agents are invoked. The pattern is: when `providerMode === 'claude-code'`, create an MCP server bound to the relevant RequestContext, create a Claude Code provider with that MCP server, and set it on the RequestContext as `'claude-code-provider'`.

Create a shared helper function in `src/mastra/workflow/steps/02-hypothesize.ts` (or inline in each step file if simpler) since the pattern repeats:

```typescript
import { createMcpToolServer } from '../../mcp/mcp-tool-bridge';
import { createClaudeCode } from 'ai-sdk-provider-claude-code';
import { CLAUDE_CODE_DISALLOWED_TOOLS } from '../../claude-code-provider';

function attachMcpProvider(
  rc: RequestContext<WorkflowRequestContext>,
  mastra: Mastra,
  providerMode: ProviderMode,
  testToolMode: 'committed' | 'draft',
): void {
  if (providerMode !== 'claude-code') return;
  const mcpServer = createMcpToolServer(rc, mastra, { testToolMode });
  const provider = createClaudeCode({
    defaultSettings: {
      disallowedTools: [...CLAUDE_CODE_DISALLOWED_TOOLS],
      permissionMode: 'bypassPermissions',
      allowDangerouslySkipPermissions: true,
      maxToolResultSize: 50000,
      mcpServers: {
        'lo-solver-tools': mcpServer,
      },
    },
  });
  rc.set('claude-code-provider', provider);
}
```

If defining this helper inline in the main step file is cleaner, do so there and import it in the sub-step files. Otherwise, create a small utility file. Use judgment (Claude's discretion per CONTEXT.md).

**File 1: `src/mastra/workflow/steps/02b-hypothesize.ts`**

In `runHypothesize`, after creating `perspectiveRequestContext` and setting all its keys, but BEFORE calling `streamWithRetry`, add:

```typescript
attachMcpProvider(perspectiveRequestContext, params.mastra, ctx.providerMode, 'draft');
```

This gives the hypothesizer agent (which uses `testRuleWithRuleset` aliased as `testRule`) the MCP server with draft-mode tester tools. The `'draft'` mode means `testRule` in the MCP server maps to the WithRuleset handler. Each perspective gets its own MCP server for state isolation (different vocabulary/rules Maps).

**File 2: `src/mastra/workflow/steps/02c-verify.ts`**

In `runVerify`, after creating `verifyRequestContext` and setting all its keys, but BEFORE calling `streamWithRetry` for the verifier-orchestrator, add:

```typescript
attachMcpProvider(verifyRequestContext, params.mastra, ctx.providerMode, 'committed');
```

This gives the verifier-orchestrator (which uses committed-rules `testRule`/`testSentence`) the MCP server with committed-mode tester tools. The `'committed'` mode means `testRule` in the MCP server maps to the committed-rules handler. Note: the verifier-feedback-extractor is tool-free, so it does NOT need MCP -- the existing singleton Claude Code provider handles it via the agent-factory fallback.

**File 3: `src/mastra/workflow/steps/02d-synthesize.ts`**

Two MCP server creation points:

1. After creating `ctx.mainRequestContext` keys setup (for synthesizer agent), before `streamWithRetry` for `hypothesis-synthesizer`, add:
   ```typescript
   attachMcpProvider(ctx.mainRequestContext, params.mastra, ctx.providerMode, 'draft');
   ```
   The synthesizer uses `testRuleWithRuleset` aliased as `testRule`, same as the hypothesizer.

2. After creating `convergenceRequestContext` and setting its keys, before `streamWithRetry` for `verifier-orchestrator`, add:
   ```typescript
   attachMcpProvider(convergenceRequestContext, params.mastra, ctx.providerMode, 'committed');
   ```
   The convergence verifier uses committed-rules `testRule`/`testSentence`.

Note: The verifier-feedback-extractor calls in both 02c and 02d do NOT need MCP providers -- they are tool-free agents.

**IMPORTANT considerations:**

1. The `attachMcpProvider` call must happen AFTER all RequestContext keys are set (especially `step-writer`, `provider-mode`, `vocabulary-state`, `rules-state`, `structured-problem`, `current-rules`) because the MCP tool handlers read from RequestContext via closure.

2. Each MCP server is lightweight (just closures over the RequestContext). Creating one per perspective is fine and necessary for state isolation.

3. Do NOT set `claude-code-provider` on RequestContexts used by tool-free agents (extractor, feedback-extractor, answerer). They use the singleton via the agent-factory fallback.

4. The `mainRequestContext` in 02d-synthesize.ts already has `event-source` set to `'merged'`. The MCP tools will emit trace events via the step-writer on this RequestContext, correctly tagged as merged events.

5. Do NOT set `continue` or `resume` on the Claude Code provider settings -- let each agent call start a fresh session.
