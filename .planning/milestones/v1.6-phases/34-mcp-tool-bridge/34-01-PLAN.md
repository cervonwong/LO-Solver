<objective>
Create the MCP tool bridge infrastructure: a factory function that wraps all 14 Mastra tools as in-process MCP tools via `createCustomMcpServer`, plus the RequestContext and agent-factory changes needed to inject per-execution Claude Code providers with MCP servers attached.

Purpose: This is the core infrastructure enabling tool-using agents to work through Claude Code. Claude Code ignores AI SDK tool definitions and only uses MCP or built-in tools, so all Mastra tools must be exposed via an MCP server.

Output: `src/mastra/mcp/` directory with MCP server factory and tool descriptions, updated `request-context-types.ts` with provider key, updated `agent-factory.ts` with per-execution provider lookup.
</objective>

**File 2: `src/mastra/mcp/mcp-tool-bridge.ts`**

Create the MCP server factory function:

```typescript
import { createCustomMcpServer, type MinimalCallToolResult } from 'ai-sdk-provider-claude-code';
import type { McpSdkServerConfigWithInstance } from '@anthropic-ai/claude-agent-sdk';
```

Export a `createMcpToolServer` function with signature:
```typescript
export function createMcpToolServer(
  requestContext: RequestContext<WorkflowRequestContext>,
  mastra: Mastra,
  options?: { testToolMode?: 'committed' | 'draft' }
): McpSdkServerConfigWithInstance
```

The `testToolMode` parameter controls which tester tool variants are registered under the `testRule` and `testSentence` names:
- `'committed'` (default): `testRule` -> uses committed rules from RequestContext, `testSentence` -> uses committed rules from RequestContext. Also registers `testRuleWithRuleset` and `testSentenceWithRuleset` as separate tools.
- `'draft'`: `testRule` -> uses WithRuleset handler (accepts ruleset as parameter), `testSentence` -> uses WithRuleset handler. Also registers `testRuleWithRuleset` and `testSentenceWithRuleset` as separate tools.

This aliasing matches what the agents see: hypothesizer/synthesizer/improver prompts reference `testRule` meaning the WithRuleset variant (they pass their draft ruleset), while verifier-orchestrator prompts reference `testRule` meaning the committed-rules variant.

Implementation approach for each MCP tool handler:
1. Construct a `ToolExecuteContext` from closure-captured `requestContext` and `mastra`:
   ```typescript
   const ctx = {
     requestContext: { get: (key: string) => requestContext.get(key as keyof WorkflowRequestContext) },
     mastra,
   } as unknown as ToolExecuteContext;
   ```
2. Call the existing Mastra tool's `execute` function with the parsed args and ctx
3. Wrap the result in `MinimalCallToolResult` format: `{ content: [{ type: 'text', text: JSON.stringify(result) }] }`
4. Catch errors and return `{ content: [{ type: 'text', text: errorMessage }], isError: true }`

Create helper functions:
```typescript
function jsonResult(data: unknown): MinimalCallToolResult {
  return { content: [{ type: 'text', text: JSON.stringify(data) }] };
}
function errorResult(message: string): MinimalCallToolResult {
  return { content: [{ type: 'text', text: message }], isError: true };
}
```

For the tool registration, use `createCustomMcpServer({ name: 'lo-solver-tools', version: '1.0.0', tools: { ... } })`. For each tool:
- `inputSchema`: Use the SAME Zod schema from the Mastra tool definition (import it). `createCustomMcpServer` expects `ZodObject` and accesses `.shape`.
- `description`: From `MCP_TOOL_DESCRIPTIONS`
- `handler`: Async function that delegates to the existing Mastra tool's execute function
- `annotations`: `{ readOnly: true }` for get* tools, `{ destructive: false, openWorld: false }` for mutation tools

Import the actual Mastra tools to call their execute functions:
```typescript
import { getVocabulary, addVocabulary, updateVocabulary, removeVocabulary, clearVocabulary, vocabularyEntrySchema } from '../workflow/vocabulary-tools';
import { getRules, addRules, updateRules, removeRules, clearRules } from '../workflow/rules-tools';
import { ruleSchema } from '../workflow/workflow-schemas';
import { testRuleTool, testRuleWithRulesetTool } from '../workflow/03a-rule-tester-tool';
import { testSentenceTool, testSentenceWithRulesetTool } from '../workflow/03a-sentence-tester-tool';
```

For each vocabulary/rules tool, the handler calls `toolInstance.execute(args, ctx)` where args matches the inputSchema and ctx is the constructed ToolExecuteContext.

For tester tools, the handler similarly delegates to the tool's execute function. The tester tools access `ctx.mastra` to call `mastra.getAgentById()` for sub-agent spawning, so the closure-captured `mastra` instance is critical.

Use tool annotations:
- `getVocabulary`, `getRules`: `{ readOnly: true, openWorld: false }`
- `addVocabulary`, `updateVocabulary`, `removeVocabulary`, `clearVocabulary`, `addRules`, `updateRules`, `removeRules`, `clearRules`: `{ destructive: false, openWorld: false }`
- `clearVocabulary`, `clearRules`: `{ destructive: true, openWorld: false }`
- All tester tools: `{ readOnly: true, openWorld: false }` (they spawn sub-agents but do not mutate state)

IMPORTANT: For tools with no input parameters (getVocabulary, clearVocabulary, getRules, clearRules), use `z.object({})` as the inputSchema.

IMPORTANT: The Mastra tool `execute` function signature is `execute(inputData, context)`. When calling from MCP handler, pass the MCP handler's `args` as `inputData` and the constructed `ToolExecuteContext` as `context`.

```typescript
if (providerMode === 'claude-code') {
```

```typescript
if (providerMode === 'claude-code') {
  // Use per-execution provider with MCP tools if available (tool-using agents)

  if (mcpProvider) {

}
```
